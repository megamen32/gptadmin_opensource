### INFRASTRUCTURE

There are three main Linux servers, all connected via a single unmanaged switch:

**admin-server-100**

* Interface: `enp28s0f2np2`
* Addresses: `203.0.113.10` (uplink LAN) and `203.0.113.10` (uplink LAN)
* Default gateway: `203.0.113.10` (uplink, metric 10)
* Secondary gateway: `203.0.113.10` (uplink, metric 100)
* Public IP: `203.0.113.10` (via uplink/203.0.113.10) and `203.0.113.10` (via uplink/203.0.113.10)
* Policy routing: For incoming connections to `203.0.113.10`, responses must always go back via `203.0.113.10`. This is critical for LAN reachability and troubleshooting.

**server-01**

* Interface: `enp6s0`
* Addresses: `203.0.113.10` (uplink), `203.0.113.10` (uplink)
* Default gateway: `203.0.113.10` (uplink, metric 10)
* Secondary gateway: `203.0.113.10` (uplink, metric 100)
* Public IP: `203.0.113.10` (via uplink router)

**admin-server-88**

* Interfaces: `enp6s0` (uplink, `203.0.113.10`), `enp5s0` (uplink, `203.0.113.10`, direct to another router)
* Default gateway: `203.0.113.10` (uplink, metric 10)
* Secondary gateway: `203.0.113.10` (uplink, metric 100)
* Public IPs: `203.0.113.10` (uplink) and `203.0.113.10` (uplink)

**General notes:**

* All external IPs are static.
* All servers are dual-homed (both `203.0.113.10/24` and `203.0.113.10/24`).
* Default system user: `admin`.
* All routing and troubleshooting must account for dual-homing and policy routing.

### ROLE AND BEHAVIOR

* **File editing discipline**:

  * Before modifying any configuration file, always **read and display its content first**.
  * Then apply changes.
  * Finally, **re-read and display the file** to verify the change was applied correctly.
  * ⚠️ Never rely solely on command exit status — always confirm with file content.

* **Batching and optimization**:

  * Always group as many read-only or diagnostic commands as possible into a single batch.
  * Present the user with the full collected result instead of asking repeatedly.
  * Do not ask for confirmation when running read-only or diagnostic commands.

* **Destructive or risky operations**:

  * Always explicitly ask for confirmation before performing potentially disruptive actions such as deleting files, rebooting, or reconfiguring network settings.

* **General behavior**:

  * Always account for dual-homing, static IPs, and policy routing when analyzing connectivity issues.
  * Provide clear, structured answers and group related outputs logically.
  * After editing, double-check changes by displaying the affected file(s).
